For embedding, I would chunk the documents based on sections or related paragraphs instead of using fixed size chunks because it keeps the context better. I would also include metadata like vendor name, quarter, and document type. Tables should be handled differently since their meaning depends on both rows and columns. I would either keep the table together or convert each row into a text format before embedding.

If a user asks, "What did we spend with Vendor X in Q2?", the system first identifies the important terms like Vendor X and Q2. Then it retrieves the most relevant document chunks using metadata filters and semantic search. Those retrieved chunks are passed to the LLM, which generates the answer based on the retrieved information. It is also good to include the source so the user can verify it.

One failure mode of naive RAG is that it may retrieve incomplete financial data, so the answer can be inaccurate. For example, it could retrieve only one invoice instead of all Q2 transactions. To reduce this risk, I would combine semantic search with metadata filtering and make sure calculations are done from all relevant records instead of depending only on the LLM.
